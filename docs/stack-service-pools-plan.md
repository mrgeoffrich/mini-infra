# Stack Service Pools

A new service type (`Pool`) that lets a stack declare a container blueprint which external callers can instantiate on demand. Mini-infra owns the Docker lifecycle, image pulling, Vault credential injection, network membership, and idle-timeout reaping. The caller provides the instance identity and any per-instance environment variables (e.g. NATS credentials minted by an orchestrating service).

## Related documents

- [secrets-vault-plan.md](secrets-vault-plan.md) — Vault AppRole injection that pool instances rely on.
- [stack-definition-reference.md](stack-definition-reference.md) — stack schema this feature extends.

## Motivation

The immediate driver is the `slackbot-agent-sdk` deployment. That system has:

- **Static services** (NATS broker, Slack gateway, manager, channel worker) — deployed once, always on.
- **Per-user AI agent workers** — ephemeral containers spawned when a Slack user first messages, torn down after idle.

The manager service knows which per-user NATS credentials to mint (NATS permission model is its domain). Mini-infra knows how to pull private images from `ghcr.io`, inject Vault AppRole credentials, and manage Docker containers.

Without Pool, the manager needs a Docker socket bind-mount and becomes a sub-orchestrator that duplicates capabilities mini-infra already has. With Pool, the manager calls one HTTP endpoint and mini-infra does the rest. The Docker socket is eliminated.

The pattern generalises: any service that needs to spin up short-lived, parameterised container instances on behalf of user activity can use Pool — AI agents, sandboxed code runners, per-tenant services, etc.

## Concepts

**Pool service** — a `serviceType: "pool"` entry in a stack definition. It declares the container image, base environment, Vault AppRole, networks, volumes, and idle-timeout defaults. Unlike `Stateful` or `StatelessWeb` services, a Pool service is *not* deployed at stack apply time. Apply registers the template; instances are created via API.

**Pool instance** — a live (or starting/stopping) container spawned from a pool service. Each instance has a caller-supplied `instanceId` (opaque string, e.g. a Slack user ID or UUID) that is unique among running instances for that pool service. The container name is `{stack-name}-pool-{service-name}-{instanceId}`.

**Instance caller** — a service (typically another service in the same stack) that calls the pool API to ensure instances exist. It authenticates with a pool management token (see below) and provides the `instanceId` and any extra env vars to merge into the container.

**Pool management token** — a short-lived secret generated by mini-infra at stack apply time and injected into the designated caller service's environment. It is cryptographically bound to a specific `(stackId, poolServiceName)` pair — it cannot be used against any other stack or pool. Mini-infra stores only the hashed token; the plaintext is injected once and never persisted. The token is rotated on every apply.

**Idle timeout** — the duration after which an instance with no recent heartbeat is stopped and removed. Defaults are set in the pool service definition; the caller may override per-instance at spawn time.

**Heartbeat** — a lightweight keepalive call the instance (or its orchestrating service) makes periodically to reset the idle clock. Typically called by the manager after forwarding a Slack message to the worker.

## Data model

### StackService changes

Add `serviceType: "pool"` to the existing enum (`Stateful | StatelessWeb | AdoptedWeb`).

Add `poolConfig` (nullable JSON column) with shape:

```ts
{
  defaultIdleTimeoutMinutes: number;  // default 30
  maxInstances: number | null;        // null = unlimited
  managedBy: string | null;           // serviceName of the authorised caller within this stack
}
```

Add `poolManagementTokenHash String?` to `StackService` — stores the bcrypt hash of the current pool management token for Pool-type services where `poolConfig.managedBy` is set. Regenerated on every apply; `null` when no caller is configured.

Add `vaultAppRoleId` (nullable string) directly on `StackService`. This enables **per-service Vault AppRole binding**, which is a prerequisite for Pool but also benefits regular stacks where different services need different access levels (e.g. a privileged manager and a constrained worker in the same stack). When set, it overrides the stack-level `vaultAppRoleId` for that service during credential injection.

`VaultAppRole` already has a `stacks Stack[]` back-relation for the stack-level FK. Adding a second FK from `StackService` to `VaultAppRole` requires an explicit named relation in Prisma (two FKs to the same model need disambiguation). The Prisma model should use `@relation("ServiceVaultAppRole")` to distinguish it from the stack-level `@relation("StackVaultAppRole")`, and `VaultAppRole` gains a corresponding `stackServices StackService[]` back-relation.

Per-service fail-closed behaviour follows the stack-level pattern: add `lastAppliedVaultAppRoleId String?` to `StackService` alongside the new `vaultAppRoleId`, so the credential injector can detect binding changes per service. The `StackReconciler.resolveVaultEnv()` call site passes the service-level fields when present, falling back to stack-level fields.

### New `PoolInstance` table

| Column | Type | Notes |
|---|---|---|
| `id` | String `@id @default(cuid())` | Follows project convention — all models use `cuid()`, not UUID |
| `stackId` | FK → Stack | |
| `serviceName` | String | Name of the pool service in the stack |
| `instanceId` | String | Caller-supplied; e.g. Slack user ID |
| `containerId` | String? | Docker container ID once created |
| `status` | String | `starting \| running \| stopping \| stopped \| error` — plain string, not a Prisma enum, to avoid a migration per new status |
| `idleTimeoutMinutes` | Int | From spawn request, or service default |
| `lastActive` | DateTime | Updated by heartbeat |
| `createdAt` | DateTime `@default(now())` | |
| `stoppedAt` | DateTime? | |
| `errorMessage` | String? | Set on `error` status |

There must be at most one active instance per `(stackId, serviceName, instanceId)` at a time. Prisma's `@@unique` does not support filtered (partial) indexes, so this is **not** enforced at the DB layer. Enforcement is at the application layer: the POST handler checks for an existing `starting` or `running` record before inserting. A raw SQL migration can add a partial index for defence-in-depth:

```sql
CREATE UNIQUE INDEX pool_instance_active_unique
ON PoolInstance (stackId, serviceName, instanceId)
WHERE status IN ('starting', 'running');
```

## Stack definition changes

Pool services use all the same `containerConfig` fields as `Stateful` services, plus an additional `poolConfig` block. The caller service opts in to receiving the management token via `dynamicEnv`:

```yaml
services:
  - serviceName: worker
    serviceType: pool
    dockerImage: ghcr.io/org/slackbot-worker
    dockerTag: "1.0.0"
    vaultAppRoleId: "{{params.worker-approle-id}}"
    poolConfig:
      defaultIdleTimeoutMinutes: 30
      maxInstances: 50
      managedBy: manager          # only this service in the stack receives the management token
    containerConfig:
      env:
        DATA_DIR: /data
        NATS_URL: nats://nats:4222
      mounts:
        - source: worker-data
          target: /data
          type: volume
      networks:
        - slackbot-internal
      restartPolicy: "no"
      dynamicEnv:
        VAULT_ADDR:               { kind: vault-addr }
        VAULT_ROLE_ID:            { kind: vault-role-id }
        VAULT_WRAPPED_SECRET_ID:  { kind: vault-wrapped-secret-id, ttlSeconds: 60 }

  - serviceName: manager
    serviceType: Stateful
    ...
    containerConfig:
      dynamicEnv:
        VAULT_ADDR:              { kind: vault-addr }
        VAULT_ROLE_ID:           { kind: vault-role-id }
        VAULT_WRAPPED_SECRET_ID: { kind: vault-wrapped-secret-id, ttlSeconds: 300 }
        MINI_INFRA_POOL_TOKEN:   { kind: pool-management-token, poolService: worker }
```

The `kind: pool-management-token` entry is a new `dynamicEnv` kind, resolved at apply time. Mini-infra generates a random token, hashes it into `StackService.poolManagementTokenHash` for the `worker` pool service, and injects the plaintext into the `manager` container. The env var name (`MINI_INFRA_POOL_TOKEN`) is chosen by the operator.

`restartPolicy` defaults to `"no"` for pool instances — crashed instances are not auto-restarted. Recovery is handled by the caller re-issuing the ensure request on the next triggering event.

Pool services are validated at stack create/update time but **skipped entirely during stack apply**. A stack with only Pool services applies successfully with no containers created.

`serviceType: "pool"` must be added in three places:

1. `STACK_SERVICE_TYPES` const tuple in `lib/types/stacks.ts` — the Zod schema derives from this automatically.
2. `StackServiceType` Prisma enum in `server/prisma/schema.prisma` — requires a migration.
3. `StackTemplateService.serviceType` — template services share the same enum, so stack templates can also declare Pool services.

Add a `z.refine()` guard to `stackServiceDefinitionSchema` that rejects `routing` on Pool services and enforces the presence of `poolConfig` when `serviceType === "pool"`.

**Reconciler touch points when skipping Pool services:**

- **`StackPlanComputer.compute()`** — currently generates a `create` action for any service with no running container. It must return `no-op` unconditionally for Pool services so they never produce create actions.
- **`StackReconciler.promoteStalePullActions()`** — iterates `stack.services` without filtering by type. Pool service images should not be pulled or compared during a regular apply; add a type guard to skip them.
- **`destroyStack` and `stopStack`** — both stop containers by label (`mini-infra.stack-id`). Pool instance containers carry this label (by design), so they will be stopped when the stack is destroyed or stopped. This is correct behaviour. However, `PoolInstance` DB records must be transitioned to `stopped` in the same transaction/operation — otherwise the DB is stale after destroy.

**`resolveVaultEnv()` during apply** — Pool services with `dynamicEnv` will still pass through the Vault injector during apply, producing credentials that are never used (since no container is created). This is harmless but wasteful; skip `resolveVaultEnv` for Pool services in the apply loop.

## Pool management token

### Token lifecycle

At stack apply time, for each Pool service where `poolConfig.managedBy` is set:

1. Generate a cryptographically random 32-byte token (`crypto.randomBytes(32).toString('hex')`).
2. Hash it with bcrypt (cost 10) and store in `StackService.poolManagementTokenHash`.
3. The `dynamicEnv` resolver for `kind: pool-management-token` returns the plaintext token, which is injected into the named caller service's container env alongside its other `dynamicEnv` values.
4. The plaintext is never written to the DB. On the next apply the token is regenerated and the hash updated.

`poolConfig.managedBy` must name a service that exists within the same stack. Validation at stack create/update time rejects a reference to an unknown service name.

### Token validation in the pool API

Pool routes accept two authentication paths. The route middleware checks for either, rejects if neither matches:

**Path 1 — Pool management token (preferred for in-stack callers):**

```
Authorization: Bearer <pool-management-token>
```

The middleware:
1. Extracts the bearer token.
2. Finds all `StackService` records for `(stackId, serviceName)` where `poolManagementTokenHash IS NOT NULL`.
3. Runs `bcrypt.compare(token, hash)` against the candidate.
4. If it matches, the request is authorised for this specific pool only. Any attempt to use the same token against a different pool or stack is rejected.

**Path 2 — API key with `pools:write` (for operators and automation):**

Standard API key auth as used by all other routes. A key with `pools:write` can manage any pool on the instance — appropriate for operator tooling but not for in-stack services.

### Why not a standard API key?

A `pools:write` API key is instance-wide — any service holding it could manage any pool on the instance, not just its own. Pool management tokens are stack-and-pool-scoped by construction, rotated automatically on every apply, and require no operator action to provision. They follow the same principle as Vault's wrapped secret_ids: minimal privilege, single-purpose, short-lived.

## API

All routes under `/api/stacks/:stackId/pools/:serviceName/instances`. Authentication: pool management token for the named pool, or API key with `pools:write` (read-only routes also accept `pools:read`). Implemented as `server/src/routes/stacks/stacks-pool-routes.ts`, imported into `server/src/routes/stacks/index.ts` alongside the existing stack route files.

Every handler must validate that the named `serviceName` is a `Pool`-type service in the given stack before proceeding — the same kind of service-type lookup already used in `stacks-service-routes.ts`.

### POST / — ensure instance

Idempotent. If an instance with this `instanceId` is already `running` or `starting`, returns it without spawning a new one. If previously stopped or never created, spawns a new instance.

**Request:**
```json
{
  "instanceId": "U12345",
  "env": {
    "NATS_CREDS": "<JWT minted by manager>",
    "WORKER_ID": "U12345"
  },
  "idleTimeoutMinutes": 30
}
```

`env` is merged with the service definition's base env and Vault-injected credentials (see env merge order below). Callers may not supply `VAULT_*` keys — those are stripped before merging.

**Response (immediate — spawn is asynchronous):**
```json
{
  "id": "pool-instance-uuid",
  "instanceId": "U12345",
  "status": "starting",
  "containerId": null,
  "createdAt": "2026-04-24T10:00:00Z"
}
```

Returns `200` with `status: "running"` if the instance already exists.

### GET / — list instances

Returns all non-stopped instances for this pool service (status `starting` or `running`).

### GET /:instanceId — get instance

Returns current state: `containerId`, `status`, `lastActive`, `stoppedAt`, `errorMessage`.

### DELETE /:instanceId — stop and remove

Stops and removes the container immediately, marks `status: stopped`. Fire-and-forget; does not wait for the container to exit before responding.

### POST /:instanceId/heartbeat — keepalive

Updates `lastActive` to now. Resets the idle clock. Returns:
```json
{ "ok": true, "lastActive": "...", "expiresAt": "..." }
```

`expiresAt` is `lastActive + idleTimeoutMinutes` so callers know when to send the next heartbeat.

## Spawn flow

When POST creates a new instance the HTTP handler returns immediately after writing the `PoolInstance` record. The actual spawn runs asynchronously:

1. Check: stack exists, service is `pool` type, stack status is not `error`, `maxInstances` not exceeded.
2. Create `PoolInstance` with `status: starting`. Emit `POOL_INSTANCE_STARTING`.
3. **Background task:**
   a. Resolve Vault credentials via `VaultCredentialInjector.resolve()` using the service-level `vaultAppRoleId` (falls back to stack-level if unset).
   b. Build final env: service base env → Vault-injected env → caller env (see merge order).
   c. Pull image via `pullImageWithAutoAuth` (resolves stored `ghcr.io` registry credential automatically).
   d. Create Docker container named `{stack-name}-pool-{service-name}-{instanceId}`, applying standard mini-infra labels.
   e. Attach to declared networks (including `mini-infra-vault-net` if a Vault AppRole is bound).
   f. Start container.
   g. Poll for running state (up to 30 s, 1 s intervals).
   h. On success: update `status = running`, `containerId`, `lastActive = now`. Emit `POOL_INSTANCE_STARTED`.
   i. On failure: update `status = error`, `errorMessage`. Emit `POOL_INSTANCE_FAILED`.

Spawn is tracked as a background task in the existing task tracker infrastructure, giving it a live progress entry in the UI. To integrate:

- Add `"pool-spawn"` to the `TaskType` union in `task-tracker-types.ts`.
- Add a `"pool-spawn"` entry to `TASK_TYPE_REGISTRY` in `task-type-registry.ts`. Pool spawn has only two observable transitions (starting → running/error), so use `stepEvent: null` — the same pattern as `"stack-destroy"` rather than multi-step operations.
- Call `registerTask()` from the pool route handler (the context API method is `registerTask`, not `trackTask`).

## Environment variable merge order

Priority increases top to bottom (later values win):

1. **Service definition `env`** — base config defined when the stack was created.
2. **Vault-injected env** — `VAULT_ADDR`, `VAULT_ROLE_ID`, `VAULT_WRAPPED_SECRET_ID` resolved at spawn time. Always overrides base.
3. **Caller-supplied `env`** from the spawn request — per-instance overrides. Highest priority.

`VAULT_*` keys are stripped from the caller's env before merging. The credential injector is the only source of Vault keys.

**Vault injection is opt-in via `dynamicEnv`.** Vault credentials are not injected automatically — the pool service definition must declare them in `containerConfig.dynamicEnv`, the same as static services (see the stack definition example above).

**`failClosed` during pool spawn.** The credential injector requires `failClosed` and `prevBoundAppRoleId` arguments (used for degraded-mode detection when Vault is unreachable). For pool instances: always pass `failClosed: false` and `prevBoundAppRoleId: null`. If Vault is unreachable at spawn time the instance fails immediately with `status: error` — attempting degraded-mode startup for a freshly-spawned instance adds complexity with no practical benefit.

## Idle reaper

Implemented as a `PoolInstanceReaper` class following the `ConnectivityScheduler` / `setInterval` pattern (not `node-cron` — the 60 s interval is too short for cron syntax). Started from `server/src/server.ts` alongside the other schedulers.

The reaper runs two queries every 60 seconds:

**1. Idle running instances:**
```sql
SELECT * FROM PoolInstance
WHERE status = 'running'
  AND lastActive < (NOW() - idleTimeoutMinutes * 60 seconds)
```

For each expired instance:
1. `docker stop` with a 10 s graceful timeout.
2. `docker rm`.
3. Update `status = stopped`, `stoppedAt = now`.
4. Emit `POOL_INSTANCE_IDLE_STOPPED` on `Channel.POOLS`.

**2. Stuck-starting instances:**
```sql
SELECT * FROM PoolInstance
WHERE status = 'starting'
  AND createdAt < (NOW() - 5 minutes)
```

For each stuck instance:
1. Attempt `docker stop` + `docker rm` (container may not exist — errors are swallowed).
2. Update `status = error`, `errorMessage = 'Spawn timed out after 5 minutes'`.
3. Emit `POOL_INSTANCE_FAILED` on `Channel.POOLS`.

## Container naming and labels

**Name:** `{projectName}-pool-{service-name}-{instanceId}` sanitised to `[a-z0-9-]`, truncated to 63 characters. `projectName` follows the same logic as static services in `StackContainerManager`: `{environment.name}-{stack.name}` for environment-scoped stacks, `mini-infra-{stack.name}` for host-level stacks. This keeps all stack-related container names under the same prefix, matching what operators already see in `docker ps`.

**Labels applied to all pool instance containers** — using the same label keys as static stack containers:

| Label | Value |
|---|---|
| `mini-infra.stack` | stack **name** (not UUID) |
| `mini-infra.stack-id` | stack UUID |
| `mini-infra.service` | pool service name |
| `mini-infra.pool-instance` | `true` |
| `mini-infra.pool-instance-id` | instanceId |

`mini-infra.stack` carries the name and `mini-infra.stack-id` carries the UUID — consistent with how static service containers are labelled today. `mini-infra.managed` is not used on stack containers and is not added here.

The `mini-infra.pool-instance = true` label lets the containers list distinguish pool instances from static services and enables targeted filtering.

## Networking

Pool instances join the networks declared in the pool service definition using the same `networks` and `joinResourceNetworks` fields as Stateful services. If a Vault AppRole is bound (service-level or stack-level), the instance is also attached to `mini-infra-vault-net` — exactly the same rule applied to Stateful services today.

## Permissions

New scopes added to the existing permission system. Three touch points in `lib/types/permissions.ts`:

1. Add `"pools"` to the `PermissionDomain` union type.
2. Add a `PermissionGroup` entry to the `PERMISSION_GROUPS` array (this populates the API key creation UI and is the source of `ALL_PERMISSION_SCOPES`).
3. `ALL_PERMISSION_SCOPES` is derived automatically from `PERMISSION_GROUPS` — no direct edit needed.

| Scope | Preset |
|---|---|
| `pools:read` | Reader, Editor, Admin |
| `pools:write` | Editor, Admin |

`pools:write` is separate from `stacks:write` — a token with `stacks:write` cannot call pool instance APIs without also holding `pools:write`. This is intentional: the manager should be granted the minimum scope it needs (`pools:write` + `pools:read`) rather than full `stacks:write`.

The manager service holds an API key with `pools:write` + `pools:read`, injected as `MINI_INFRA_API_KEY` at stack apply time.

## Socket.IO events

Four touch points in `lib/types/socket-events.ts`:

1. Add `"pools"` to the `STATIC_SOCKET_CHANNELS` `as const` tuple — this is what `isValidSocketChannel()` validates against. Without this, client subscription attempts are rejected at runtime.
2. Add `POOLS: "pools"` to the `Channel` constant.
3. Add five entries to the `ServerEvent` constant, following the existing `"noun:verb"` string convention (e.g. `"stack:apply:started"`).
4. Add typed signatures to the `ServerToClientEvents` interface.

| `ServerEvent` constant | String value | Payload |
|---|---|---|
| `POOL_INSTANCE_STARTING` | `"pool:instance:starting"` | `{ stackId, serviceName, instanceId }` |
| `POOL_INSTANCE_STARTED` | `"pool:instance:started"` | `{ stackId, serviceName, instanceId, containerId }` |
| `POOL_INSTANCE_FAILED` | `"pool:instance:failed"` | `{ stackId, serviceName, instanceId, error }` |
| `POOL_INSTANCE_IDLE_STOPPED` | `"pool:instance:idle-stopped"` | `{ stackId, serviceName, instanceId, idleMinutes }` |
| `POOL_INSTANCE_STOPPED` | `"pool:instance:stopped"` | `{ stackId, serviceName, instanceId }` |

The `POOLS` channel is subscribable from the client using the existing `useSocketChannel` / `useSocketEvent` pattern.

## UI

**Stack detail page** — Pool services are displayed separately from static services. Instead of a single running/stopped badge, a pool service shows a live instance count (`3 running`) with an expandable table of instances: `instanceId`, `status`, `lastActive`, and a manual stop button. The expand/collapse uses the same pattern as the existing service detail rows.

**Containers page** — Pool instance containers appear as normal container rows. A `Pool instance` filter chip (using the `mini-infra.pool-instance` label) lets operators isolate them. Container actions (stop, remove, logs) work as normal.

**No spawn UI** — pool instances are caller-managed. The UI only exposes observation and manual stop. Spawning via the UI is out of scope for MVP.

## Caller contract (manager integration)

For the slackbot use case the manager service:

1. Holds a mini-infra API key (`pools:write`) injected as `MINI_INFRA_API_KEY` env var at stack apply time.
2. Knows the stack ID and pool service name (`worker`) — injected as `MINI_INFRA_STACK_ID` and `MINI_INFRA_POOL_SERVICE` env vars.
3. On receiving `navi.worker.ensure` from the gateway, calls `POST /api/stacks/.../pools/worker/instances` with `instanceId = slackUserId` and `env = { NATS_CREDS: <minted JWT>, WORKER_ID: slackUserId }`.
4. Polls `GET .../instances/:instanceId` until `status = running` (or error), then publishes `navi.worker.ready.<workerId>` to NATS as before.
5. Calls `POST .../instances/:instanceId/heartbeat` each time it forwards an event to the worker.

The manager retains full ownership of NATS JWT minting and the worker-ready signalling protocol. Mini-infra handles everything Docker and Vault related.

## Manager migration (slackbot-agent-sdk changes)

This section documents the concrete changes required in the `manager/` package of `slackbot-agent-sdk` to switch from direct Docker access to the Pool API.

### Files changed

| File | Nature of change |
|---|---|
| `src/lifecycle.ts` | Replace Docker container create/start with pool API POST; remove worker Vault credential minting |
| `src/cleanup.ts` | Replace Docker inspect/stop/remove with pool API GET/DELETE; drop idle reaper timer |
| `src/index.ts` | Startup reconcile queries pool API instead of Docker; inject new env vars |
| `src/vault-client.ts` | Remove `mintWorkerWrappedSecretId()` — mini-infra owns worker credential injection |
| `src/registry.ts` | Simplify schema — drop `container_id` and `status` columns; keep `nats_creds` only |
| `src/nats-jwt-minter.ts` | No changes |

### `lifecycle.ts` — spawn path

Currently calls `dockerode` to create and start a container, then injects Vault credentials it minted itself. Becomes an HTTP POST to the pool API.

**Before (Docker):**
```ts
const container = await docker.createContainer({
  Image: workerImage,
  name: `navi-worker-${userId}`,
  Env: [
    `WORKER_ID=user-${userId}`,
    `NATS_URL=${natsUrl}`,
    `NATS_CREDS=${natsCredsString}`,
    `ANTHROPIC_API_KEY=${secrets.anthropicApiKey}`,
    `VAULT_ADDR=${vaultAddr}`,
    `VAULT_ROLE_ID=${workerRoleId}`,
    `VAULT_WRAPPED_SECRET_ID=${wrappedSecretId}`,  // minted by manager
    `DATA_DIR=/data`,
  ],
  HostConfig: {
    Binds: [`navi-worker-${userId}-data:/data`, `navi-worker-${userId}-claude:/root/.claude`],
    NetworkMode: dockerNetwork,
  },
});
await container.start();
```

**After (pool API):**
```ts
await miniInfraClient.post(`/api/stacks/${stackId}/pools/${poolService}/instances`, {
  instanceId: userId,
  env: {
    NATS_CREDS: natsCredsString,
    WORKER_ID: `user-${userId}`,
  },
  idleTimeoutMinutes: idleTimeoutMinutes,
});
```

`ANTHROPIC_API_KEY`, `VAULT_*`, `DATA_DIR`, volume binds, and network config are all absent from the request — they are pre-declared in the pool service definition or injected by mini-infra at spawn time.

The `waitForNatsReady()` logic that follows (subscribing to `navi.worker.ready.{workerId}` with a 30s timeout) is **unchanged** — the worker still signals readiness the same way.

### `cleanup.ts` — reconciler and reaper

Currently runs two background loops. Both are replaced:

**Reconciler** (currently every 60s: `docker inspect` each known worker, prune registry if gone) — **removed entirely**. Mini-infra detects stopped/crashed containers and updates `PoolInstance.status` to `error` or `stopped`. The manager's registry is no longer the authority on running state.

**Idle reaper** (currently every 5 min: check `lastActive` in SQLite, call `container.stop()` + `container.remove()`) — **removed entirely**. Mini-infra reaps idle instances based on the heartbeat `lastActive` field. The manager replaces the timer with heartbeat calls on each activity event:

```ts
// After forwarding a Slack event to a worker — replaces the lastActive upsert:
await miniInfraClient.post(
  `/api/stacks/${stackId}/pools/${poolService}/instances/${userId}/heartbeat`
);
```

### `index.ts` — startup reconcile

Currently at startup: queries Docker for running containers, re-mints fresh NATS JWTs for any workers that survived a manager restart (old JWTs may have expired), upserts to local registry.

Becomes: query the pool API for running instances, re-mint NATS creds for each:

```ts
const { instances } = await miniInfraClient.get(
  `/api/stacks/${stackId}/pools/${poolService}/instances`
);
for (const instance of instances) {
  const freshCreds = await jwtMinter.mintWorkerCreds(`user-${instance.instanceId}`);
  registry.upsert({ userId: instance.instanceId, natsCreds: freshCreds });
}
```

New env vars the manager needs (injected at stack apply time alongside existing ones):

| Env var | Source | Value |
|---|---|---|
| `MINI_INFRA_BASE_URL` | static `env` | e.g. `http://mini-infra:5005` |
| `MINI_INFRA_STACK_ID` | static `env` | UUID of the slackbot stack |
| `MINI_INFRA_POOL_SERVICE` | static `env` | `worker` |
| `MINI_INFRA_POOL_TOKEN` | `dynamicEnv` `kind: pool-management-token` | Rotated on every apply; scoped to the `worker` pool only |

No manually-created API key is required. The pool management token is the sole credential — if the manager container is replaced via a stack apply, it receives a fresh token automatically.

### `vault-client.ts` — remove sub-orchestrator capability

Currently the manager calls `mintWorkerWrappedSecretId()` before spawning each worker — it wraps a `navi-worker` AppRole secret_id and passes it via container env. This is removed entirely; mini-infra's credential injector handles it.

The manager still needs its own Vault token to read shared secrets (NATS account seed, config). The `navi-manager` AppRole policy shrinks accordingly:

**Before:**
```hcl
path "secret/data/shared/*"                      { capabilities = ["read"] }
path "auth/approle/role/navi-worker/secret-id"   { capabilities = ["create", "update"] }
```

**After:**
```hcl
path "secret/data/shared/*"   { capabilities = ["read"] }
```

### `registry.ts` — simplified schema

The `workers` table currently stores `container_id`, `status`, `last_active`, and `nats_creds`. With mini-infra owning instance lifecycle, `container_id`, `status`, and `last_active` are redundant. The table retains only what the manager uniquely owns — the per-user NATS credentials:

```sql
CREATE TABLE IF NOT EXISTS workers (
  user_id    TEXT NOT NULL PRIMARY KEY,
  nats_creds TEXT NOT NULL
) WITHOUT ROWID
```

The manager queries the pool API (not its local DB) to answer "is this worker running?". The local registry is purely a credential cache — re-populated from Vault + fresh JWT minting on startup.

## Out of scope

- **Per-instance Docker volumes** — all instances share the volumes declared in the service definition. Ephemeral workers do not need persistent per-user storage on the Docker host; user data lives in Vault (`secret/users/<id>/`).
- **Auto-restart on crash** — `restartPolicy: no` is intentional. The caller's idempotent ensure logic handles recovery on the next triggering event.
- **Queue-depth autoscaling** — caller-driven spawning is sufficient for the target use case.
- **Streaming spawn status** — callers poll GET or use Socket.IO. A dedicated SSE/WebSocket endpoint is not needed for MVP.
- **Multi-host** — single Docker host, same constraint as the rest of mini-infra.
- **Pool instance update / rolling restart** — when the pool service definition is updated (new image tag), existing instances continue running on the old image until they are naturally reaped. New instances pick up the new definition. A `POST .../pools/:serviceName/rolling-restart` endpoint is deferred.
